## 1. 这份文档的目标

前面的文档已经把产品结构、页面职责和前端实现边界讲清楚了。

这份文档专门解决两个最容易被忽视、但最影响“顺不顺手”的问题：

1. 不同页面和不同任务状态下，界面到底该怎么响应
2. 文案到底该怎么说，用户才会觉得清楚、直观、没有技术噪音

换句话说，这份文档负责定义：

- 空状态
- 运行状态
- 异常状态
- 可导出状态
- 不可导出状态
- 推荐动作
- 按钮文案
- 提示文案
- 术语统一

## 2. 文案与交互的总原则

## 2.1 文案原则

### 原则 A：先说结果，再说原因

不要直接说技术事实：

- `OCR subtitles missing`

要先说用户关心的结果：

- `当前无法导出英文字幕版，因为还没有 OCR 英文字幕。`

### 原则 B：先说“能做什么”，再说“为什么”

不要只告诉用户哪里有问题，要同时告诉用户下一步。

不好：

- `当前未识别到可预览字幕文件。`

更好：

- `当前还没有可用于烧录的英文字幕。你可以改用 ASR 字幕，或补跑 OCR 翻译。`

### 原则 C：避免工程术语直出

用户默认不应该看到：

- template
- run_to_stage
- task-g
- subtitle_source
- delivery policy

用户应该看到：

- 成品目标
- 执行范围
- 英文字幕来源
- 最终成片内容

### 原则 D：一个文案只表达一个判断

不要在同一行混合多个抽象概念。

不建议：

- `当前 OCR 缺失且 clean video unavailable, fallback asr is recommended`

建议拆开：

- `当前没有 OCR 英文字幕。`
- `当前也没有干净画面。`
- `你可以先改用 ASR 字幕，或补跑缺失素材。`

### 原则 E：状态标签短，说明文案完整

标签用短词：

- 运行中
- 已完成
- 可导出
- 已导出
- 受阻

说明文案可以更完整。

## 4. 全局状态模型

## 4.1 任务级状态

| 状态 | 面向用户标签 | 说明 |
|---|---|---|
| `pending` | 排队中 | 已创建，尚未开始 |
| `running` | 运行中 | 正在执行 |
| `succeeded` | 已完成 | 执行完成 |
| `partial_success` | 部分完成 | 核心链路完成，但有缺失 |
| `failed` | 失败 | 执行中断或关键步骤失败 |

## 4.2 导出级状态

建议前端抽象出单独的导出状态，而不是只复用任务状态。

| 状态 | 面向用户标签 | 说明 |
|---|---|---|
| `not_ready` | 待补齐素材 | 当前不能导出目标版本 |
| `ready` | 可导出 | 当前素材已满足导出条件 |
| `exporting` | 导出中 | 正在生成预览或成品 |
| `exported` | 已导出 | 至少已生成一份成品 |
| `blocked` | 受阻 | 当前导出目标被特定素材缺失阻塞 |

## 4.3 素材级状态

| 状态 | 面向用户标签 | 说明 |
|---|---|---|
| `available` | 可用 | 素材存在且可用 |
| `missing` | 缺失 | 当前没有该素材 |
| `building` | 生成中 | 上游任务正在产出 |
| `failed` | 失败 | 曾尝试生成但失败 |

## 6. 任务列表页状态与文案

## 6.1 全列表为空

- 标题：`还没有任务`
- 说明：`创建一个任务后，这里会显示运行状态、导出状态和历史记录。`
- 主按钮：`新建任务`

## 6.2 搜索无结果

- 标题：`没有找到匹配的任务`
- 说明：`试试修改搜索词，或清空当前筛选条件。`
- 次按钮：`清空筛选`

## 6.3 列表行的状态表达

建议每一行至少有两个状态块：

1. 任务状态
2. 导出状态

示例：

- `运行中` + `—`
- `已完成` + `可导出`
- `已完成` + `待补齐素材`
- `失败` + `—`

## 6.4 行内动作文案

| 条件 | 主动作 | 次动作 |
|---|---|---|
| 运行中 | 查看详情 | 停止 |
| 失败 | 重跑 | 查看详情 |
| 已完成且可导出 | 导出 | 查看详情 |
| 已完成但受阻 | 补齐素材 | 查看详情 |
| 已导出 | 再次导出 | 查看详情 |

## 7. 新建任务页状态与文案

## 7.1 页面标题与步骤名称

建议替换当前步骤名称：

当前：

- 基础信息
- 节点配置
- 高级选项
- 确认提交

建议改为：

- 素材与语言
- 成品目标
- 质量与设置
- 确认创建

原因：

- 更符合用户决策顺序
- 降低工程术语比例

## 7.2 输入视频为空

### 状态文案

- 提示：`请输入本地视频文件路径，或选择一个视频文件。`

### 主 CTA

- `下一步`

### CTA 状态

- 禁用

## 7.3 视频探测成功

### 状态文案

- `已识别视频信息，可继续选择成品目标。`

### 辅助信息

- 时长
- 分辨率
- 格式

### 不建议文案

- `probe success`

- `如果路径正确但仍失败，可能是文件格式异常或没有读取权限。`

### 主 CTA

- `重新检测`

## 7.8 创建前摘要文案模板

建议统一使用：

```text
本次任务将生成：
- {结果类型}
- 语言：{源语言} → {目标语言}
- 默认导出：{导出摘要}
- 系统将自动启用：{关键能力摘要}
```

英文字幕版示例：

```text
本次任务将生成：
- 英文字幕版
- 语言：中文 → 英文
- 默认导出：优先干净画面 + 英文字幕
- 系统将自动启用：OCR 字幕链路、导出预览能力
```

- `服务暂时不可用，请稍后重试。`

## 14. i18n 改造建议

## 14.1 现有文案问题

当前 [messages.ts](/Users/masamiyui/OpenSoureProjects/Forks/video-voice-separate/frontend/src/i18n/messages.ts) 中还保留了很多偏技术的表述，例如：

- `节点配置`
- `展示字幕来源`
- `交付视频底板`
- `交付音轨`
- `交付策略`

这些建议逐步替换为用户语言。

## 14.2 建议新增的文案分组

建议新增：

- `taskIntent`
- `exportWizard`
- `assetStatus`
- `exportReadiness`
- `recommendations`

这样可以避免把“导出相关文案”继续塞进 `taskDetail` 或 `newTask` 的旧结构里。

## 14.3 文案组织建议

每个状态尽量成组配置：

- 标题
- 说明
- 主 CTA
- 次 CTA

不要只在代码里到处拼字符串。

## 15. 最终结论

如果前面的结构设计解决的是“页面怎么分工”，那么这份文档解决的是“用户在每一步到底读到什么、点什么、怎么继续往下走”。

这次体验要真正顺起来，必须同时做到两件事：

1. 页面分工清楚
2. 文案和状态表达也清楚

只改结构、不改状态文案，用户仍然会觉得别扭。

最关键的判断标准不是页面多漂亮，而是用户在任何时刻都能一眼回答这三个问题：

1. 现在是什么状态
2. 为什么会这样
3. 我下一步该点什么
