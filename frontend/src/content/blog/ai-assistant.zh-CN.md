# 把一句话变成一条调用链——会编排原子能力的 AI 助手

> 这是 **幕后 · 博客** 的一篇**架构**。上一篇[不碰算法的流水线编排器](/blog/pipeline-orchestration)，讲的是一条**预先排好**的固定链路；这一篇聊它的另一面——你随口说一句话，由 AI 助手**现场**把它编排成一条原子能力调用链。同样是"编排"，一个是套餐，一个是你点菜、它替你点单。

## 先分清：套餐、零件、和点单的人

translip 里其实有三层"干活"的东西，容易混，先掰开。

**套餐，是流水线。** 上一篇讲的编排器跑的就是它——一条 `separation → transcription → … → render` 预先排好的固定链路，适合"标准配音"这种成熟、稳定的流程。你给一个视频，它从头跑到尾。

**零件，是原子工具。** 这是和流水线**正交**的另一套系统：十几个**各干一件小事**的独立工具——分离人声、转写、语种识别、翻译、合成、配音渲染、字幕识别、字幕烧录、字幕擦除、探测媒体信息……每个都能单独上传、单独跑、单独下载产物。它们不预设任何顺序，等着被自由组合。

**点单的人，就是这篇的主角——AI 助手。** 它站在"原子工具"这一侧，干的事只有一件：**把你一句自然语言，编排成一条原子工具的调用链**。你说「把这个视频配成英文配音」，它知道这背后是 分离 → 转写 → 翻译 → 合成 → 回贴，五步，缺一不可——然后替你把这五步排好、串好、跑完。

它一个采样点都不处理，一行配音算法都不碰。和流水线编排器一样，它只负责**调度**；真正干活的，永远是底下那些既有的本地能力。区别只在于：流水线的链是写死的，助手的链是**临时、按你这句话现编的**。

## 第一步：把一句话，变成一份结构化的计划

你在对话框里打的那句话，发到后端的 `/plan`。规划这一步交给 DeepSeek——规划器（`planner.py`）带着一份**工具目录**（`catalog.py`，列着现在有哪些原子能力、各自吃什么参数）去问大模型：用户想要这个结果，请用这些工具排一条调用链出来。

关键在于，它返回的**不是一段话**，而是一个**结构化的计划**。这份计划长这样（前端的类型定义，和后端 `models.py` 一一对应）：

```ts
// 每一步，是一次明确的工具调用
interface PlanStep {
  tool_id: string                  // 用哪个原子工具
  title: string
  rationale: string                // 为什么需要这一步（给你看的理由）
  params: Record<string, unknown>  // 这一步的参数
  inputs: Record<string, Binding>  // 输入从哪来（见后文）
}

// 一份计划 = 一句话总结 + 一组步骤 + 步骤之间的连线
interface AssistantPlan {
  summary: string
  steps: PlanStep[]
  edges: StepEdge[]                // 步骤先后 —— 本质是一张有向图
}
```

注意那个 `edges`：计划不是简单的"第一步、第二步、第三步"一条直线，而是一张 **DAG**。多数时候确实是条直链，但比如"既要配音、又要烧字幕"，后半段就可能分叉。用图来表达，比硬掰成一条线**诚实**——这点和流水线那张节点依赖图是一个思路。

把"模糊的意图"翻译成"明确的、带参数和连线的步骤图"，是助手最核心的一跳。后面所有的事——展示、编辑、执行——都建立在这份结构化计划之上。

## 一个有意思的岔路：它可能先反问你

但如果你那句话本身就**没说清**呢？「给这个视频配音」——配成哪国语言，你没讲。

这时候助手不会瞎猜一个日语给你跑下去。`/plan` 的返回类型，本来就留了两条路：

```ts
// 规划的结果，可能是一份计划，也可能是一个澄清问题
interface PlanResult {
  type: 'plan' | 'clarification'
  plan?: AssistantPlan | null
  clarification?: Clarification | null  // { question: string; options: string[] }
}
```

走到 `clarification` 这条路，它回你的就不是计划，而是一句反问——"配成哪种语言？"——下面挂着中 / 英 / 日几个选项气泡，等你点。

这是刻意留的一条岔路，背后的判断是：**一个会编排的助手，最怕的不是不会干，是会错意。** 你想要中文配音，它默默跑了一版英文——二十分钟、几个重模型的算力，就这么没了。所以"先问一句"虽然多一次来回，挡住的却是"自信地跑错"。一步到位的前提是它真懂你；不懂还硬上，那不叫聪明。

## 计划不是黑箱：它摊开给你改

第二道闸，是计划**可编辑**。

助手排好的那份 `AssistantPlan`，不是个一点就跑的黑箱，而是**摊在你面前的一排可视化卡片**：每张卡片就是一个 `PlanStep`——用什么工具、吃什么输入、参数多少、为什么需要它（那句 `rationale`），写得清清楚楚。执行前，你可以逐步改参数、调顺序，甚至加一步或删一步。比如它排好了配音，你临时想"再加上把字幕烧进去"，直接在计划尾巴上追一步就行。

所以这是一种**"AI 起草、你定稿"**的关系：规划交给模型，但**拍板权在你**。它和上一节的"先反问"是一对的——前面用澄清挡住误解，这里用可编辑计划兜住"模型偶尔排得不够好"。两道闸，都是为了让你别在跑了半天之后，才发现方向错了。

## 产物怎么接上的：一条叫 Binding 的引线

回到那条 分离 → 转写 → 翻译 → 合成 → 回贴 的链。一个很实在的问题：第一步分离出人声，第二步转写要拿这个人声当输入——这个"产物的传递"，在计划里是怎么表达的？

答案是 `PlanStep` 里那个 `inputs`。它的值不是写死的文件路径，而是一条 `Binding`——说清楚"我这个输入从哪来"：

```ts
// 每个输入，是一条"从哪来"的引线
interface Binding {
  source: 'upload' | 'step'  // 来自你最初的上传，或某个前置步骤的产物
  upload_index?: number | null
  step_id?: string | null    // 来自哪一步
  output?: string | null     // 要那一步的哪个产物
}
```

来源就两种：要么 `upload`——你最开始拖进来的那个原始视频；要么 `step`——某个**前置步骤的输出**，记着它的 `step_id` 和具体要哪个产物。于是转写那一步的输入，就是一条指向"分离步骤的人声输出"的 Binding。

整条链，本质就是**一串 Binding 把上一步的产物，接到下一步的输入上**。这层抽象最舒服的地方是：作为用户，你**全程没碰过一个文件**——中间产物叫什么、存哪儿、谁喂给谁，全被 Binding 这层引线吃掉了，助手顺着它们自动转运。最后所有中间产物加最终成片，都能一键下载，那些被接来接去的文件，一个都没丢。你只看到"目标进、成片出"。

## 执行：每一步，其实是一个原子任务

确认计划后，它发到 `/execute`。执行器（`executor.py`）按那张图的顺序，**一步一步把每个 step 变成一个原子工具任务**丢下去跑。

这里有个和流水线编排器同源的脾气：**它没有另起一套执行引擎**。它复用的，就是原子工具本来那套作业系统——有并发队列、能取消、产物会登记。所以助手里的每一步，在运行状态里都对应一个 `job_id`，跟你手动用单个工具跑出来的，是**同一条路**：

```ts
// 运行时，每一步的实时状态
interface RunStepState {
  id: string
  tool_id: string
  job_id?: string | null     // 它就是一个原子工具作业
  status: string
  progress_percent: number   // 实时进度
  error_message?: string | null
  artifacts: StepArtifact[]  // 这一步产出的、可下载的文件
}
```

你盯着进度时看到的，就是这些：在跑哪一步、百分之多少、日志、以及这步产出的文件——前端靠 SSE 把它们实时推上来，所以那条进度条是真在动。哪一步出错了，错误信息也单独挂在那一步上，不会糊成一团。

"复用既有作业系统"听着像偷懒，其实是 translip 一以贯之的克制：**助手不发明新的执行能力，它只是编排既有的。** 能被编排进来的，永远是仓库里已经有的那些本地工具——这条边界很重要，下一节正好聊到。

## 跑完不丢：沉淀成一条「AI 任务」

一次对话不是聊完就散。每一次规划 + 执行，都会**落库成一条「AI 任务」**。

它和流水线任务、原子任务，是任务中心里**平级的三类记录**——只是 AI 任务多了一层来历："它是被一句话派生出来的"。后端给它配了一整套生命周期：

```text
POST /assistant/plan            一句话 → 计划 / 或澄清问题
POST /assistant/execute         执行计划，起一次运行
GET  /assistant/runs            AI 任务列表
GET  /assistant/runs/{id}       某次运行的实时状态
POST /assistant/runs/{id}/cancel   取消
POST /assistant/runs/{id}/rerun    重跑
DELETE /assistant/runs/{id}        删除记录
```

所以你能在 AI 任务列表里回看每一次：用了哪些工具、几步、完成几步、跑了多久、成没成；能复制这份计划再次执行、取消正在跑的、删掉不要的，甚至导出审计日志。聊天是入口，但每一次"聊"的背后，是一次**可追溯、可复跑**的运行。

## 边界与那条底线

最后说清楚它的边界——尤其有一条，关乎你的素材会不会被传走。

**第一，它要配 `DEEPSEEK_API_KEY` 才会规划。** 把"一句话拆成调用链"这件事，是云端大模型干的；没配 key，助手就罢工。这个在「全局设置」里填。

**第二，也是那条底线：只有"规划"这一步会上云。** 发给 DeepSeek 的，**仅仅是你那句话和那份工具目录**——你的媒体内容，视频、音频，**不出本地**。所有真正干活的原子能力，分离、转写、翻译、合成、擦除、视觉感知，全是在你机器上跑的。换句话说，云端只参与"排个计划"，一帧画面、一段声音都不会经过它。

**第三，它只会编排已经存在的东西。** 它不会凭空变出一个仓库里没有的能力；模型偶尔也会把计划排得不够好、或者会错意——所以前面那两道闸（**先反问澄清**、**计划可编辑**）才是必需品，不是装饰。

## 几个「为什么这么做」

把这一层的取舍收拢一下：

- **自然语言入口**：你描述目标，不用先认识十几个工具、几十个参数。"选工具、调参数、接产物"这些琐碎，助手替你扛。
- **规划交给模型、拍板交给你**：DeepSeek 出结构化计划，但计划**可澄清、可编辑**——两道闸挡住"会错意"和"排得不够好"，让你别跑到一半才发现方向错。
- **产物用 Binding 接力**：每一步的输入是一条"从哪来"的引线，而不是写死的路径；文件搬运被这层抽象吃掉，用户全程不碰文件。
- **复用既有作业系统**：每一步就是一个原子工具作业（带 `job_id`、进度、产物），助手不造新引擎，只编排既有能力。
- **本地干活、只有规划上云**：媒体内容不出本地，这是写进设计里的底线，不是事后补的承诺。

## 小结

这位 AI 助手，做的是一件很"产品"、却也很"架构"的事：把一句模糊的人话，落成一条**你能看、能改、能复跑**的原子能力调用链，中间所有的文件搬运它替你扛，最后沉淀成一条带履历的 AI 任务——而它自己，始终是个不碰算法、只负责把活儿派明白的"会编排的同事"。

它和上一篇的流水线编排器，是同一种克制的两个面：一个把固定流程编排成一台可恢复的机器，一个把自由组合编排成一句话就能用的对话。下一篇，我们回到算法线，继续 **Task C：翻译**——看一段转写好的文字，是怎么被"看着画面、记着是谁说的"翻成另一种语言的。
